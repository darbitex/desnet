## 1. `desnet::assets` — fractal-tree on-chain binary upload

### Why

Mint events embed media references via `MintMedia { kind, mime, ref_blob_id, ... }`.
For media >8KB (the inline-payload cap), creators need a place to put the
bytes themselves. Off-chain backends (Walrus, Arweave, IPFS) work but break
the "everything verifiable on-chain" property of DeSNet. `assets` is the
first-class **on-chain** backend (BACKEND_DESNET_ASSETS = 3).

### What it does

Stores arbitrary binary blobs up to **5 MB** as a fractal tree of 30 KB
chunks — small enough to fit each chunk in a single Aptos tx, no
network-level fragmentation issues.

```
Master ──┬── Node ──┬── Chunk[0..29 KB]
         │          ├── Chunk[0..29 KB]
         │          └── Chunk[0..29 KB]
         └── Node ──┬── Chunk[...]
                    └── ...
```

Tree shape:
- **depth 0**: 1 chunk only (≤30 KB blob)
- **depth 1**: 1 root node holding N chunks (≤30 N B blob)
- **depth 2**: 1 root node holding M leaf nodes, each holding C chunks
  (≤30 × M × C B blob, capped at 5 MB)

MIME validated: `PNG=1, JPEG=2, GIF=3, WebP=4, SVG=5`. SVG included for
on-chain generative-art ethos (XSS responsibility delegated to frontend img
sandboxing).

### Sealed-after-finalize

Once `finalize` is called, the Master + chunks become immutable. Any future
`deploy_chunk_for(master)` aborts. Mint events that reference a sealed
Master can rely on the bytes being permanent.

### Three deploy tiers (orchestrator_tier = 3 on mainnet)

| Tier | Entry suffix | Address allocation | Use case |
|------|--------------|---------------------|----------|
| **1** (B1) | (default) | server allocates Object addr at `start_upload`, returns to caller | simple uploader, one tx at a time, address known after first tx |
| **2** (B2) | `*_pub` | same as Tier-1 but returns the addr explicitly via return value (script-friendly) | scripts that need to chain `start_upload` → `deploy_chunk` in one tx |
| **3** (B3) | `*_v2` | **deterministic** addr derived from `(uploader, master_nonce, chunk_index)` via sha3 + `create_named_object` | frontend pre-computes EVERY address in JS before any tx fires; supports parallel chunk uploads + retry-with-known-addr |

`view::orchestrator_tier()` returns the highest tier supported (3 on
mainnet); frontends key off this to choose which entry suffix to call.

### Auth

Only the original uploader (the wallet that called `start_upload`) can
deploy chunks/nodes against that Master, and only that wallet can finalize.
`creator_pid` is informational metadata for indexers (which DeSNet PID owns
the source content); it is **not** an auth gate, so engagement-side
modules don't need PID-resolve before reading `creator_pid_of`.

## 2. `desnet::opinion` — perpetual-no-settle prediction-market substrate

### Why

DeSNet is a social network. Posts (mints) want a *belief signal* attached.
Twitter-style polls don't expose financial truth. Polymarket-style oracle
+ settle markets don't fit the "perpetual feed" paradigm — they expire.

The opinion pool gives every mint an optional **always-open belief market**:
trade YAY/NAY positions backed by the creator's own factory token, with
the market never settling, no oracle, no expiry. Belief intensity is
expressed as price; price changes propagate back to the social feed via
the standard mint+history surface.

### Architecture

When `mint::create_opinion_mint` is called, atomically:

1. `do_create_mint(...)` — same as regular `create_mint`, allocates a seq,
   emits MintEvent into history with VERB_MINT
2. `opinion::bootstrap_market_for_mint(author, author_pid, seq, initial_mc)` —
   creates an OpinionMarket at deterministic address derived from
   `(author_pid, seq)`. Uses caller's `$creator_token` (factory-spawned
   handle token) as collateral.

Pool invariant ("Mirror-Mint Bootstrap"):

```
At creation:
  vault = initial_mc creator_token
  pool_yay = initial_mc YAY (newly minted)
  pool_nay = initial_mc NAY (newly minted)
  creator gets 0 of either
  → vault == total_yay_supply == total_nay_supply == initial_mc
```

After every trade op (deposit/swap/redeem), the conservation invariant
`vault_balance == fungible_asset::supply(YAY) == fungible_asset::supply(NAY)`
is asserted on chain (`assert_conservation`).

### Trade verbs

- `deposit_pick_side(side, amount)` — pull `amount` creator_token from user,
  mint pair → user keeps chosen side, pool gets opposite side. Vault grows.
- `deposit_balanced(amount)` — pull `amount` creator_token, mint pair →
  user gets BOTH sides 1:1. Pool unchanged. Atomic balanced exit primitive.
- `swap_yay_for_nay(amount_in, min_out)` / `swap_nay_for_yay(amount_in, min_out)` —
  trader swaps within the pool via x*y=k. Vault unchanged. **Tax** =
  `0.1%` of spot-price-equivalent creator_token, burned via `apt_vault::burn_via_vault`.
- `redeem_complete_set(amount)` — burn `amount` YAY + `amount` NAY from user,
  return `amount` creator_token from vault minus tax skim. Always-exit
  guarantee for users holding balanced pairs.

### Detection (Pattern B compat-safe design)

Indexers detect opinion-mints by calling the view function:

```move
opinion::market_exists(author_pid: address, seq: u64): bool
```

Returns true iff an `OpinionMarket` resource exists at the deterministic
addr derived from `(author_pid, seq)`. There is **no `is_opinion` flag in
MintEvent** (preserves v0.3.3 BCS layout for compat-safe upgrade).

### Composability

Opinion-mints ARE regular mints — `MintEvent` is identical, written to the
same history stream with `VERB_MINT`. So all 7-verb reactions
(spark/voice/echo/remix/press/sync) work natively on opinion-mints. The
opinion AMM is a parallel feature layered on top, not a separate post type.

### Constants (locked, not parameterized)

- `MIN_INITIAL_MC = 1M` whole tokens (`1e14` raw at 8dec)
- `MAX_INITIAL_MC = 100M` whole tokens (`1e16` raw)
- `DEFAULT_TAX_BPS = 10` (0.1%) — fixed, defended against drift via
  `E_TAX_DRIFT=16` assert in `burn_tax`
- `MAX_OPINIONS_PER_PID = 10,000` — anti-grief cap

## Module dependency graph (additions)

```
mint  ──→ opinion::bootstrap_market_for_mint  (friend, called atomically inside create_opinion_mint)
mint  ──→ assets::is_sealed / mime_of         (existing)

opinion ──→ profile::derive_pid_signer        (friend)
opinion ──→ apt_vault::burn_via_vault         (friend, tax burn destination)
opinion ──→ history::append (verb_opinion=7)  (friend)
opinion ──→ factory::owner_has_token / token_metadata_of_owner (public views)
```

All friend grants are minimally scoped — `apt_vault::burn_via_vault` is
direction-locked by the FA framework (cannot burn DESNET from a
creator-token vault), `profile` friend surface is read-only / signer-derivation
only, `history::append` for trade-event emission with VERB_OPINION = 7.

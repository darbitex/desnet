## 3. 8-dimension source audit

### 3.1 ABI surface

- All 17 module ABIs match v0.3.1 source ✓
- No public/friend fn removed vs v0.3.0.6 (confirmed by re-running compat-check tool: see §6 detection method)
- v0.3.1 additions: `governance::DESNET_FA_ADDR` const, `governance::E_NEUTERED=22`, `governance::E_ARGS_LEN_MISMATCH=21` (already added in v0.3.0.6 chunked infra), `giveaway::settle_bounty_bps()` view (compat-fix added during deploy)
- v0.3.1 body changes: `governance::voting_power` (hardcoded FA addr + object_exists guard), `governance::update_desnet_fa_metadata` (neutered to abort), `governance::propose_upgrade` (dropped vestigial `desnet_fa_metadata != @0x0` check)

### 3.2 Argument validation

- `register_handle`: handle len 1..16 enforced via `validate_handle`, avatar_b64 ≤ AVATAR_MAX_BYTES, bio ≤ BIO_MAX_BYTES, reserved-handle gate per `reserved_handle_claimer(handle)` ✓
- `update_fee_receiver` / `rotate_admin`: `assert!(addr != @0x0)` ✓ (R1 fixes preserved)
- `governance::multisig_*`: vector length parity check `code_indices.len == code_chunks.len` (E_ARGS_LEN_MISMATCH) ✓
- `update_total_30d_emission`: NO upper-bound check on `amount` — multisig can set any u64 (acknowledged: deferred to v0.3.2 auto-tracker)

### 3.3 Math

- `voting_power` returns `min(earned, balance)` — both u64, no overflow ✓
- `proposal_threshold_amount` / `quorum_amount` — `total_30d_emission * BPS / 10000`. With max u64 input, `* 7000` could overflow if `total_30d_emission > u64_max/7000 ≈ 2.6e15`. Realistic emission ≤ 1B raw (= 1e16 per token at 8dec) per pool. **Per-pool risk**: if multisig sets `total_30d_emission` to a 32B+ raw value (beyond physical possible), overflow. Practically bounded by economic constraint, not contract. Recommend: sanity-cap at 1e18 in v0.3.2.
- `giveaway::settle` bounty math: `(remaining * 5) / 10000`. Max remaining bound by pool budget (≤ 1B raw token = 1e8 native). `1e8 * 5 = 5e8` < u64 max. ✓
- AMM CPMM math: V3-style split-fee bucket. Reviewed in R3, no changes in v0.3.1.

### 3.4 Reentrancy

- `governance::multisig_publish_chunked_upgrade` consumes UpgradeStaging via `move_from`, calls `code::publish_package_txn`. `publish_package_txn` is framework-blessed and atomic; cannot re-enter governance. ✓
- `register_handle` calls cross-module: `factory::create_token_atomic`, `lp_staking::create_pool_and_lock`, etc. All friend-only callees with no signer-controlled callbacks. ✓
- `giveaway::settle` does multiple `primary_fungible_store::withdraw/deposit` on same FA. No re-entry surface (FA store ops are atomic). ✓

### 3.5 Edge cases

- **L2 (LOW)**: `multisig_publish_chunked_upgrade` doesn't validate that all module slots in `code` are non-empty before publish. If a chunk is skipped (out-of-order user error), the resulting `code: vector<vector<u8>>` may have empty entries, causing `code::publish_package_txn` to fail at framework level. This will surface as a generic VM error, not a clear ours-error. Mitigation: chunker.py orders sequentially → user error not contract bug. Recommend add `assert!(vector::all(&code, |c| !vector::is_empty(c)))` defense-in-depth in v0.3.2.
- **L3 (LOW)**: `cleanup_upgrade_staging` has no `multisig_upgrade_disabled` gate — even after backdoor disabled, multisig can still call it. Acceptable: cleanup is state-reset only, doesn't publish code. No exposure. Document as intended behavior.
- `voting_power`: returns 0 pre-DESNET-FA-spawn (before `register_handle("desnet")`). Correct — voting_power is meaningless before token exists.

### 3.6 Cross-module interactions

- `governance::voting_power` reads `voter_history::rewards_earned_30d` + `primary_fungible_store::balance`. Both view-only on framework + voter_history (no recursion). ✓
- `register_handle` ordering: validation → fee-debit → PID create → factory atomic → pool atomic → stake atomic → handle map. Each step's failure aborts whole tx (Move atomicity). ✓
- The bootstrap `publisher::take_cap_for_desnet` is consumed at `init_module` — single-shot, never re-derivable. ✓

### 3.7 Errors

- **L1 (LOW) — VERIFIED IN SMOKE**: 5 factory view fns (`get_token_record`, `handle_of_token`, `handle_of_owner`, `token_metadata_of_owner`, `lp_staking_pool_of_owner`) abort with `E_HANDLE_TAKEN` when the semantic is "record not found". Confusing for frontend — user calls `handle_of_token(token_addr)` to check ownership, gets back an error literally meaning the opposite. Suggest renaming to `E_TOKEN_NOT_FOUND` (=19, already exists) or introduce `E_NOT_FOUND` const for clarity. Body change only — compat safe in v0.3.2.
- All R1 zero-addr fixes preserved (Gemini MED) ✓
- `update_desnet_fa_metadata` aborts E_NEUTERED unconditionally — non-multisig callers can also waste gas to trigger this. Cost-bounded (~50 gas), not a DOS vector. Acceptable.

### 3.8 Events

- `MultisigUpgrade` event emitted on chunked publish only (not on intermediate stage_chunk). Reasonable.
- No event for `cleanup_upgrade_staging` — stale state cleared silently. Recommend add `UpgradeStagingCleanup { multisig, timestamp_secs }` for observability in v0.3.2.
- `register_handle` emits via factory (TokenSpawned), AMM (PoolCreated), lp_staking (PositionCreated). No standalone `HandleRegistered` event in profile.move — recommend add for indexability in v0.3.2.

## 6. Compat-violation detection method (recovered the hard way)

Detection of the R3-baseline → v0.3.1 compat violation that aborted the first chunked deploy attempt (VMError 1096 BACKWARD_INCOMPATIBLE_MODULE_UPDATE):

```python
# For each module in publish_payload.json:
#   1. Extract bytecode hex → bytes → /tmp/mod_NN.mv
#   2. Disassemble: aptos move disassemble --bytecode-path
#   3. Grep `^public fun` and `^friend fun` from .masm
#   4. Fetch /v1/accounts/<@desnet>/module/<name>.abi.exposed_functions
#   5. Filter chain ABI by visibility ∈ {public, friend}
#   6. Diff: chain_set - payload_set = MISSING (compat violation), payload_set - chain_set = NEW (additive, OK)
```

Detected `giveaway::settle_bounty_bps` MISSING in v031 payload — the R3 baseline lacked it, but on-chain v0.3.0.5/0.3.0.6 had added it. Adding it to source as a 1-line view fn returning the const fixed the compat. Keep this routine as pre-deploy gate for future upgrades.
